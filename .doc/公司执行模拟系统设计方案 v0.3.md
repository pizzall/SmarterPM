# 公司执行模拟系统设计方案 v0.3

---

## 一、系统定位

本系统为**任务人选推荐工具**，核心目标是：

- 在指定的组织结构范围内，为负责人推荐最合适的任务执行人选组合
- 提供可解释的推荐理由，支持负责人二次纠正
- 随纠正记录动态更新人员能力模型
- 字段缺失时通过逐级推断保证候选人不被排除

系统**不负责**任务跟进、进度管理、绩效考核。

---

## 二、数据结构设计

### 2.1 组织结构（JSON）

组织以**双层结构**存储：行政树 + 虚拟项目组，两者共享人员节点。

```json
{
  "org": {
    "id": "company",
    "name": "公司",
    "children": [
      {
        "id": "dept_product",
        "name": "产品部",
        "head": "emp_001",
        "children": [
          {
            "id": "dept_design",
            "name": "设计小组",
            "head": "emp_003",
            "children": []
          }
        ]
      },
      {
        "id": "dept_tech",
        "name": "技术部",
        "head": "emp_002",
        "children": []
      }
    ]
  },
  "project_groups": [
    {
      "id": "proj_001",
      "name": "跨端重构项目组",
      "head": "emp_002",
      "members": ["emp_001", "emp_005", "emp_008"],
      "status": "active"
    }
  ]
}
```

**结构要点：**

- 每个节点含 `head`（负责人）字段，负责人可向下递归调用所有子节点人员
- 跨部门调用需经由共同上级节点，系统在推荐时会标注调用路径和审批层级
- 虚拟项目组与行政树并列存储，人员可同时属于多个节点

### 2.2 人员档案（JSON）

所有字段除 `id`、`name`、`departments` 外均为**可选字段**，缺失时由 AI 按推断规则处理。

```json
{
  "id": "emp_005",
  "name": "张三",
  "departments": ["dept_tech", "dept_product"],
  "role_tendency": "executor",
  "mbti": "INTJ",
  "skills": [
    { "tag": "后端开发", "level": 4 },
    { "tag": "系统架构", "level": 3 },
    { "tag": "Python",   "level": 5 }
  ],
  "work_scope": ["后端服务", "API设计", "数据库优化"],
  "communication": 3,
  "responsibility": 4,
  "growth_rate": 3,
  "performance_trend": "stable",
  "collaboration_notes": [],
  "correction_log": [],
  "special_notes": null
}
```

#### 字段说明

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|----------|------|
| `departments` | 数组 | ✅ | 支持兼任多部门 |
| `role_tendency` | 枚举 | 可选 | `leader` / `executor` / `reviewer` |
| `mbti` | 字符串 | 可选 | 用于AI推断团队适配性，缺失时由其他字段估算 |
| `skills` | 数组 | 可选 | 层级技能标签 + 1-5分等级，缺失时由工作范围推断 |
| `work_scope` | 数组 | 可选 | 工作范围描述，技能缺失时的第一推断来源 |
| `communication` | 1-5 | 可选 | 沟通能力，缺失时使用部门均值估算 |
| `responsibility` | 1-5 | 可选 | 工作负责度，缺失时使用部门均值估算 |
| `growth_rate` | 1-5 | 可选 | 学习/成长速度 |
| `performance_trend` | 枚举 | 可选 | `rising` / `stable` / `declining`，由负责人输入更新 |
| `collaboration_notes` | 数组 | 可选 | AI与负责人共同维护的协作记录 |
| `correction_log` | 数组 | 自动 | 历次纠正记录，用于追溯能力值变化及隐性能力推断 |
| `special_notes` | 字符串 | 可选 | 管理者自由填写的特殊备注，如特殊约束、临时状态说明等，不参与评分计算，仅供推荐时参考展示 |

## 三、人员评价模型

### 3.1 评价维度总览

| 维度 | 来源 | 权重类型 |
|------|------|----------|
| 技能标签匹配度 | 人员档案 × 任务需求 | 硬性筛选 |
| 工作范围匹配度 | 人员档案 × 任务描述 | 硬性筛选 |
| 业务能力（综合技能等级） | 档案 | 核心评分 |
| 沟通能力 | 档案 | 辅助评分 |
| 工作负责度 | 档案 | 辅助评分 |
| 成长速度 | 档案 | 辅助评分 |
| 角色倾向 × 任务角色需求 | 档案 × 任务配置 | 结构匹配 |
| MBTI 团队适配性 | AI 推断 | 组合评分 |
| 近期绩效趋势 | 负责人输入 | 动态修正 |

### 3.2 MBTI 团队适配性（AI自动推断）

AI根据候选人的MBTI组合，在生成推荐方案时自动评估整体团队的协作风险，**无需人工配置协作矩阵**。

推断逻辑如下：

1. **功能互补性**：团队中是否同时覆盖"思考型（T）"与"情感型（F）"视角；"判断型（J）"与"感知型（P）"的节奏是否平衡
2. **主导风格冲突检测**：若团队中出现多个强主导型人格（如多个ENTJ/ESTJ），系统会标注潜在的话语权冲突风险
3. **任务类型适配**：创意类任务偏向NP型组合；执行落地类任务偏向SJ型组合；协调类任务需要F型成员参与
4. **内外向比例**：纯内向或纯外向团队分别在客户沟通和内部协作上存在风险，AI会给出提示

推断结果以**团队适配评语 + 风险标注**呈现，而非单一数值，便于负责人理解和纠正。

## 四、冷启动方案

### 4.1 设计原则

冷启动的核心不是识别"新人"，而是**让 AI 在字段缺失的情况下依然能够推断出该人员的行为逻辑**，并将其纳入正常推荐流程，不附加任何身份标签。

**缺什么字段，就用剩余字段推断；推断越充分，说明越具体。** 人员档案不做"是否完整"的分类，仅记录字段是否存在，AI 自行决定每个字段的处理方式。

### 4.2 字段缺失时的逐级推断规则

**技能字段缺失**
- 有 `work_scope`：AI 将工作范围文本映射到技能标签做近似匹配，标注"技能来自工作范围推断"
- 有岗位职责描述但无 `work_scope`：以职责描述做宽泛匹配，说明匹配依据较粗
- 两者均无：该维度跳过，不影响其他维度参与评分

**MBTI 缺失**
- 有 `role_tendency` + `communication` 评分：AI 从行为字段推断大致的 I/E 和 T/F 倾向，用于团队适配性评估，标注"MBTI 由行为字段估算"
- 仅有岗位描述：AI 根据岗位典型特征做弱推断（如"架构师"偏 INT 倾向），标注"MBTI 由岗位类型估算"
- 均无：团队适配性评估中跳过该成员，不影响其余成员的适配分析

**能力评分缺失（communication / responsibility）**
- 有 `correction_log`：从历史纠正记录中提取隐含的能力评价（如曾被替换并注明原因），作为能力参考
- 无任何记录：以同部门同级别人员的均值作为临时估算值，标注"使用部门均值估算"
- 均值也无法计算（部门过小）：该字段跳过评分，仅参与定性描述

**全字段缺失（仅有姓名 + 部门）**
- AI 以部门属性推断可能的工作范围，将该人员以低权重纳入候选池
- 推荐理由仅描述"部门与任务存在方向重叠"，不输出具体评分
- 与其他人员一视同仁地呈现在方案中，负责人可自行判断是否采用

### 4.3 推荐呈现方式

推断依据在推荐理由中**自然附带说明**，而非用独立标签区分人员身份：

```
候选人：李四（产品部）
匹配依据：工作范围描述与任务需求重叠（前端开发方向）
          沟通能力使用部门均值估算（3.0）
          MBTI 缺失，团队适配性未纳入计算
推荐理由：范围匹配度较高，其余维度数据待积累，建议负责人结合实际了解判断
```

不出现"新人""不确定性"等标签，仅描述**具体哪些依据充分、哪些依据是估算的**，判断权留给负责人。每次任务参与后，负责人的纠正输入会自动补充该人员的数据积累，推断链条逐渐被真实数据替代。

## 五、任务配置与推荐流程

### 5.1 任务结构

```json
{
  "id": "task_042",
  "title": "用户中心模块重构",
  "description": "将现有用户系统拆分为微服务架构，涉及API重设计与数据库迁移",
  "requester": "emp_001",
  "complexity": "epic",
  "required_roles": {
    "leader": 1,
    "executor": 3,
    "reviewer": 1
  },
  "required_skills": ["系统架构", "后端开发", "数据库优化"],
  "duration_weeks": 4,
  "sprint_id": "sprint_07",
  "primary_task": "完成用户认证服务的微服务拆分与新旧接口平滑切换",
  "sub_tasks": [
    "迁移现有数据库表结构，保证数据一致性",
    "编写接口对接文档，同步前端团队"
  ]
}
```

#### 主次任务字段说明

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|----------|------|
| `primary_task` | 字符串 | 可选 | 本次任务的核心目标，单条自由文本；用于在推荐理由中对齐人员能力与任务重点 |
| `sub_tasks` | 数组 | 可选 | 次要任务条目列表，可为空；每条为独立字符串，描述附带交付项或辅助工作 |

主次任务的设定为推荐引擎提供更精细的需求对齐依据：`primary_task` 优先驱动技能匹配，`sub_tasks` 辅助评估人员的宽度覆盖能力。

### 5.2 任务复杂度与人员搜索范围

| 复杂度 | 中文 | 第一搜索域 | 扩展域 | 最终域 |
|--------|------|-----------|--------|--------|
| `normal` | 普通 | 同部门直接下属 | — | — |
| `advanced` | 高级 | 同部门全员 | 上级部门成员 | — |
| `epic` | 史诗 | 同部门全员 | 上级部门成员 | 全公司 |

搜索域按顺序逐级展开：**优先在最小范围内满足需求，不足时再扩展**，扩展行为在推荐结果中明确标注。

### 5.3 推荐输出格式

AI输出**2-3套候选方案**，每套方案包含：

```
【方案 A】优先匹配能力

成员列表：
- 王五（技术部）· 角色：Leader · 综合评分：4.2
  匹配理由：系统架构 Lv4，近期绩效上升，INTJ 适合主导复杂规划型任务
- 张三（技术部）· 角色：Executor · 综合评分：3.8
  匹配理由：后端开发 Lv4，Python Lv5，与王五历史协作记录良好
- 李四（产品部）· 角色：Executor
  匹配理由：工作范围与前端需求重叠；沟通能力使用部门均值估算（3.0），MBTI 未纳入计算
- 赵六（技术部）· 角色：Reviewer · 综合评分：3.5
  匹配理由：数据库优化 Lv4，沟通能力 4，适合审查角色

团队适配性评语：
  王五（INTJ）+ 张三（INTP）整体偏内向思考型，执行精度高但客户沟通
  可能偏弱，建议确保赵六承担对外协调职责。李四 MBTI 数据缺失，未纳入组合评估。

方案优势：核心能力覆盖完整
方案风险：李四部分数据依赖估算，外部沟通稍弱
跨部门调用：李四需产品部负责人（emp_001）确认
```

## 六、负责人纠正与能力值更新

### 6.1 纠正流程

```
AI输出推荐方案
        ↓
负责人审阅（可接受 / 部分调整 / 完全推翻）
        ↓
负责人可输入：
  · 替换某成员，并说明理由（如"张三近期状态差"）
  · 对某成员的特定维度给出评价（如"王五沟通比评分显示的强"）
  · 对整体方案的定性评价（如"这种类型任务不适合内向型主导"）
        ↓
AI根据输入执行定标更新
```

### 6.2 能力值更新规则

| 输入类型 | 更新方式 |
|----------|----------|
| 具体维度评价 | 直接微调对应字段（±0.5，不超出1-5范围） |
| 定性评价 | AI解析意图，映射到最相关维度并更新，同时记录原始输入 |
| 绩效趋势描述 | 更新 `performance_trend` 字段，影响下次推荐权重 |
| 人选替换理由 | 写入被替换人员的 `correction_log`，作为下次推荐参考，同时为缺失字段补充估算依据 |

**所有更新均追加写入 `correction_log`，不覆盖原始值**，保证可追溯：

```json
{
  "date": "2026-04-29",
  "task_id": "task_042",
  "field": "communication",
  "old_value": 3,
  "new_value": 3.5,
  "source": "负责人输入：王五在复盘会上表现超预期",
  "updated_by": "emp_001"
}
```

## 七、冲刺周期管理

```json
{
  "sprint_id": "sprint_07",
  "start_date": "2026-04-28",
  "duration_weeks": 2,
  "tasks": ["task_042", "task_043", "task_044"]
}
```

每个冲刺周期结束时，系统生成**简报视图**：

- 本周期任务列表 + 推荐方案是否被采纳
- 被纠正的推荐记录（帮助识别系统的系统性偏差）
- 能力值发生变化的人员清单

简报主要服务于负责人回顾，**不做自动评估或评分**。

## 八、技术实现要点

### 数据存储
- 组织结构、人员档案、任务记录均以 **JSON 文件**本地存储
- 按 `org.json` / `employees/` / `tasks/` / `sprints/` 分目录管理

### AI 模型接入
- 任务描述 → 技能需求提取：由 AI 解析非结构化任务文本，自动映射到技能标签
- 候选人评分：基于加权规则计算，AI 负责 MBTI 适配推断、字段缺失推断和定性评语生成
- 纠正意图解析：将负责人的自然语言输入解析为字段级更新操作

### 人员数据初始化
- 最小录入：姓名 + 部门，系统即可将其纳入推荐候选池
- 其余字段均为可选，缺失字段由 AI 按推断规则处理，不影响入选资格
- 随每次任务参与和负责人纠正记录逐步填充完整，能力模型自然演化

### 任务回顾页面行为
- 提交回顾后页面**原地重新渲染**（不触发整页刷新、不依赖 hash 跳转），历史回顾列表即时更新
- 同次回顾提交同步触发能力值变更提案生成，Toast 提示条数后跳转至"能力值待审"页面处理

## 九、核心设计原则汇总

| 原则 | 体现 |
|------|------|
| 推荐可解释 | 每个人选附带具体匹配理由和推断依据说明 |
| 方案可对比 | 输出多套方案而非单一最优 |
| 数据可追溯 | 所有纠正写入日志，不覆盖原值 |
| 缺失可推断 | 字段缺失时逐级使用其他字段估算，任何人都不被排除 |
| 人工优先 | AI 推荐，人决策，系统记录 |
