# SmarterPM 用户体验改进计划

> **文档目的**：从非技术 PM 经理视角分析系统当前缺失和体验问题，为后续优化提供依据
> 
> **分析时间**：2026-05-13
> 
> **目标用户**：不懂技术的 PM 经理

---

## 一、核心问题总结

### 1.1 系统定位局限性

根据设计文档，系统**明确声明"不负责任务跟进、进度管理、绩效考核"**。

但从 PM 视角看，这是**致命缺陷**：
- 推荐人选只是工作的开始（约 5%）
- 执行过程管理才是 PM 每天 95% 的工作
- 系统在推荐阶段有价值，但在后续全流程中几乎没用

### 1.2 数据录入门槛过高

系统假设用户能正确填写数据，但**非技术 PM 在以下场景处处碰壁**：
- JSON 格式输入（技能、角色需求）
- ID 记忆负担（部门 ID、员工 ID）
- 枚举值输入（复杂度、状态、趋势）
- 数值范围不明（1-5 的标准是什么）
- 逗号分隔多值（中文逗号问题）

**一句话总结**：数据录入是系统工作的第一步，这一步就劝退了 80% 的目标用户。

## 二、PM 真实工作流分析

模拟一个真实的 PM 工作日：

```
09:00  PM 登录系统，看今天要处理的事
       ❌ 没有"待办视图"，不知道哪个任务需要先处理
       
09:30  想规划一个新任务"支付系统重构"
       ✅ 可以通过对话规划
       ❌ 不知道谁有空（没有实时状态）
       
10:00  AI 推荐 3 个方案
       ✅ 推荐质量不错
       ❌ 不知道推荐的赵祉皓已经在忙 2 个任务了
       
10:30  选择方案，需要通知成员
       ❌ 系统没有通知功能，PM 手动微信沟通
       
11:00  赵祉皓回复"下周请假"
       ❌ 系统不知道这个信息
       ❌ PM 需要手动重新生成方案
       
14:00  想看进行中任务的进度
       ❌ 系统不跟踪执行，需要打开 Project/Jira
       
16:00  想回顾赵祉皓过去表现
       ⚠️ 有数据，但看不到可视化总结
       
17:00  下班前想看明天要处理的任务
       ❌ 没有"任务看板"或"日历视图"
```

## 三、功能缺失分析

### 3.1 严重缺失（PM 每天都需要的）

| 功能 | 场景描述 | 现状 | 影响 |
|-----|---------|-----|-----|
| **人员实时状态面板** | PM 需要知道谁有空、谁在忙什么、谁下周请假 | 只有 `performance_trend` 字段 | 推荐的人可能已满负荷或即将休假 |
| **多任务资源冲突检测** | 同时规划 3 个任务，AI 都推荐了同一个人 | 不考虑跨任务冲突 | PM 需手动协调，系统反而制造混乱 |
| **任务执行过程跟踪** | 任务开始后，需要知道进度、延期风险、卡点 | 只管推荐，不管执行 | PM 需用其他工具跟进（Project/Jira/Excel） |
| **通知与沟通机制** | 方案确定后，通知候选人、跨部门审批 | 完全没有通知机制 | PM 手动微信/邮件沟通，推荐方案确定后脱离系统 |
| **任务状态看板** | 看当前有哪些进行中、待启动、已完成 | 只有 status 字段，无视图 | 无法快速了解全局 |

### 3.2 中等缺失（影响决策质量）

| 功能 | 场景描述 | 现状 | 影响 |
|-----|---------|-----|-----|
| **成本/预算考量** | 老板说预算有限，别用太高级的人 | 完全不考虑成本 | 无法做成本约束下的推荐 |
| **任务优先级和紧急度** | 有 10 个待规划任务，哪个先做？ | 无优先级字段 | 无法排序和优先处理 |
| **人员意愿度** | AI 推荐某人，但他对这类任务兴趣不大 | 不考虑意愿 | 可能分配意愿不强的人 |
| **任务依赖关系** | task_042 需要先等 task_041 完成 | 无依赖关系字段 | 无法批次规划 |
| **历史项目可视化对比** | 想看某人在类似任务中的历史表现 | 有数据但无可视化 | 无法快速评估匹配度 |

### 3.3 轻度缺失（用户体验）

| 功能 | 现状 | 影响 |
|-----|-----|-----|
| **Sprint 简报视图** | 有数据结构，无前端展示 | 设计文档提到但未实现 |
| **决策辅助仪表盘** | 方案对比纯文本 | 无法快速对比差异 |
| **搜索和筛选** | 没有全局搜索 | 无法快速找到信息 |
| **批量操作** | 只能单个操作 | 效率低 |
| **权限管理** | 没有权限控制 | 部门主管无法限制访问范围 |
| **移动端适配** | 未针对移动优化 | 手机上使用困难 |

## 四、体验细节问题（数据录入层面）

### 4.1 JSON 格式：非技术用户的噩梦

| 输入场景 | 当前实现 | PM 困惑 | 应该怎么做 |
|---------|---------|---------|-----------|
| **技能录入** | `[{"tag":"产品设计","level":4.0}]` | 完全看不懂格式 | 技能选择器：下拉选技能 + Slider 选等级 |
| **角色需求** | `{"leader":1,"executor":2}` | 不知道怎么写 JSON | 角色配置器：每个角色 + 数量输入框 |
| **整体编辑** | "JSON 原文" tab | 更不敢碰 | 保留但标注"高级用户"，默认隐藏 |

**真实场景**：
```
PM 想给赵祉皓加技能"前端开发 Lv4"

❌ 现在：手动输入 JSON → 输错符号 → Toast"JSON 解析失败" → 不知道哪里错
✅ 应该：点击"添加技能" → 下拉选"前端开发" → Slider 拖到 4 → 保存
```

### 4.2 应该有下拉框的字段（ENUM）

#### 完全没做下拉的：

| 字段 | 可选值 | 当前实现 | 应该怎么做 |
|-----|-------|---------|-----------|
| **角色倾向** | leader / executor / reviewer | 手动输入文本 | 下拉单选 |
| **绩效趋势** | rising / stable / declining | 手动输入文本 | 下拉单选 |
| **复杂度** | normal / advanced / epic | 手动输入文本 | 下拉 + 每项说明 |
| **任务状态** | draft / active / done / archived | 手动输入文本 | 下拉单选 |
| **回顾情绪** | positive / neutral / negative | 手动输入文本 | 下拉（中文：积极/中性/消极） |
| **MBTI** | 16 种类型 | 手动输入文本 | 下拉 + 类型说明 |
| **项目组状态** | active / inactive | 手动输入文本 | 下拉单选 |

**真实场景**：
```
PM 想设置任务复杂度为"高级"

❌ 现在：手动输入 "advanced"（要记住英文）→ 可能输入"高級"、"Advanced" → 全错
✅ 应该：下拉选择：普通 / 高级 / 史诗 + 每个选项带说明
```

#### 应该有智能提示的：

| 字段 | 当前实现 | 应该怎么做 |
|-----|---------|-----------|
| **部门选择** | 手动输入 "dept_product" | 下拉选择（显示名称 + ID） |
| **员工选择** | 手动输入 "emp_001" | 下拉选择（显示姓名 + ID）+ 搜索 |
| **技能选择** | 手动输入技能标签 | 下拉选择（技能库）+ 可新增 |
| **发起人** | 手动输入 emp_id | 下拉选择员工列表 |
| **Sprint ID** | 手动输入 sprint_id | 下拉选择 Sprint |

### 4.3 数值输入：没有范围提示

| 字段 | 范围 | 当前实现 | 应该怎么做 |
|-----|-----|---------|-----------|
| **沟通能力** | 1-5 | 手动输入数字 | Slider + 标注（1=很差 5=很好） |
| **责任度** | 1-5 | 手动输入数字 | Slider + 标注 |
| **成长速度** | 1-5 | 手动输入数字 | Slider + 标注 |
| **技能等级** | 1-5 | JSON 数字 | Slider + 标注 |
| **周期（周）** | 1-N | 手动输入 | 数字输入 + 范围提示 |

### 4.4 多值输入：逗号分隔不友好

| 字段 | 当前实现 | 问题 |
|-----|---------|-----|
| **部门** | "dept_product,dept_bfb775b5" | 不知道有哪些部门，不知道 ID |
| **工作范围** | "产品规划,需求评审" | 不知道应该填什么 |
| **技能需求** | "游戏开发,UI设计" | 不知道技能库，中文逗号问题 |
| **项目组成员** | "emp_001,emp_005" | 不知道员工 ID |

**真实场景**：
```
PM 想设置任务需要的技能

❌ 现在：手动输入 "游戏开发,UI设计"
        可能输入中文逗号"，" → 解析失败
        不知道系统有哪些技能标签

✅ 应该：多选下拉或标签选择器
        显示技能库，点击选中 → badge 显示
        可搜索/过滤
```

### 4.5 ID 记忆负担：PM 不应该背 ID

| 场景 | 当前需要输入 | PM 困惑 |
|-----|------------|--------|
| **选择部门** | "dept_product" | 什么是 dept_product？是产品部？ |
| **选择员工** | "emp_001" | emp_001 是谁？张袁圆？王翰林？ |
| **设置负责人** | "emp_002" | 不知道谁是 emp_002 |
| **设置发起人** | "emp_001" | 不知道员工 ID 列表 |
| **跨部门调用说明** | 手动写审批路径 | 不知道组织结构 |

### 4.6 输入校验和提示缺失

| 问题 | 现状 | 应该做 |
|-----|-----|-------|
| **部门不存在** | 提交后报错 | 下拉选择或实时校验 |
| **员工不存在** | 提交后报错 | 下拉选择或实时校验 |
| **数值超出范围** | 输入 6 后提交才报错 | 输入时限制范围 |
| **JSON 格式错误** | "JSON 解析失败" | 实时校验 + 指出错误位置 |
| **枚举值错误** | 输入"高級"后报错 | 下拉选择或实时提示 |
| **空值提交** | 提交后报错 | 必填标注 + 提交前校验 |

## 九、实施建议

### 9.1 分阶段实施

**第一阶段（2周）**：解决 P0 级别的数据录入问题
- 技能选择器
- 角色配置器
- ENUM 下拉
- ID 选择下拉
- Slider 控件

**第二阶段（1周）**：解决 P1 级别的体验改进
- 实时校验
- 字段说明
- 方案成员显示姓名

**第三阶段（2周）**：解决 P2 级别的功能缺失
- 人员状态面板
- 冲突检测
- 任务看板

### 9.2 技术建议

- **前端**：保持纯 JS，但引入组件化模式（可考虑 Vue/React 单文件）
- **数据结构**：在 database.json 增加 `skill_library` 和 `scope_library`
- **后端**：增加 `/api/enums` 接口返回所有 ENUM 定义
