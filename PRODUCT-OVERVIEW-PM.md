# SmarterPM 产品介绍（面向产品经理 / 业务负责人）

本文面向 **产品经理、项目管理、人力资源伙伴或业务负责人**：说明 SmarterPM「是什么」「能解决什么问题」，并按 **功能模块** 解释 **设计目的** 与 **实际用处**。技术部署请参阅 [DEPLOYMENT-PM.md](DEPLOYMENT-PM.md)，功能清单摘要请参阅 [README.md](README.md)。

---

## 一句话定位

**SmarterPM 是一套跑在本地的「公司执行模拟系统」**：你在系统里维护组织结构与人员档案，用自然语言描述任务需求，系统结合规则与可选的大模型能力，**推荐多套人选组合方案**、给出可解释的理由；任务执行后可以 **回顾**，并把共识沉淀为 **能力模型的变更提案**，形成「规划 → 派工 → 复盘 → 更新人才画像」的闭环。

它不替代真实的项目管理工具（如 Jira）或 HRIS，更适合作为 **排兵布阵与人岗匹配的沙盘**：用于方法论演练、立项前的人选辩论、培训演示，或对「理想情况下该怎样组队」做结构化推演。

---

## 核心价值（推销视角）

| 价值点 | 说明 |
|--------|------|
| **从模糊需求到结构化任务** | 用对话澄清任务边界，减少口头需求在传递中失真。 |
| **多套方案 + 理由** | 不是只有一个「正确答案」，而是 2～3 套组合，附带匹配逻辑、优势与风险，便于评审与取舍。 |
| **可解释的推荐** | 推荐与人岗字段挂钩；字段不全时系统会 **逐级推断**，尽量减少「莫名其妙被排除」的情况。 |
| **纠正即学习（模拟意义上）** | 负责人对方案的修正会记入纠正日志，并可触发 **能力值变更提案**，体现「用人反馈反哺画像」。 |
| **资源与负载可视** | 人员负载、跨任务冲突、Sprint 简报与通知，帮助 PM 从「只看任务列表」升级到「看人」与「看约束」。 |
| **数据在自己机器上** | 业务数据集中在单个 `database.json`，便于备份、迁移与演示脱敏；可选接入兼容 OpenAI 协议的 API，**无 Key 时仍可完整走通演示流程**。 |

## 系统模块总览（与界面对应）

以下模块与顶部导航、任务子页面及悬浮「自由对话」一致，按 **业务链路** 而非技术目录组织。

```
组织与人员底座 → 任务与排期视图 → 任务规划 / 人选推荐 → 方案确定与冲突检查
      → 执行期看板与 Sprint → 任务回顾 → 能力值待审 → 通知与全局助手
```

---

## 各模块：目的与用处

### 1. 组织管理（`#/org`）

**目的**：建立模拟公司的 **部门、项目组** 等结构边界，让人与任务都有「归属」，后续推荐与筛选在指定组织范围内进行。

**用处**：先做 org chart 式的配置，避免「全公司乱搜人」；与真实组织对齐时，更有利于和干系人用同一套语言讨论「在哪个部门找人」。

---

### 2. 人员档案（`#/employees`）

**目的**：维护 **员工档案**——技能、角色倾向、沟通与执行等维度（以及成本率等扩展字段），作为推荐算法的输入；对 **缺失字段** 标注推断来源，保证透明度。

**用处**：把「这人能不能干」从主观印象转成 **结构化画像**；录入层已表单化（技能编辑器、枚举说明等），降低手工维护 JSON 的负担。

---

### 3. 人员状态（`#/people-status`）

**目的**：从「静态档案」切换到 **当前负载视角**：谁在哪些任务上、并发任务数量等，并用视觉提示标示过载风险。

**用处**：PM 在做新任务规划或审批方案时，可以快速回答：**这人是否已经太忙**，避免纸上谈兵式的完美组队。

---

### 4. 任务（`#/tasks`）

**目的**：任务的 **增删改** 与详情维护（优先级、预算上限、进度、阻塞项、里程碑、依赖等），与后续规划、方案、回顾共用同一数据源。

**用处**：任务不只是标题描述，而是承载 **约束与上下文**（例如预算上限会影响推荐时对高成本角色的软降权）；详情页可进入 **方案**、**回顾** 等下游流程。

---

### 5. 看板（`#/board`）

**目的**：以看板形式查看任务状态，支持 **拖拽更新状态**，贴近常见敏捷协作心智模型。

**用处**：在执行仿真阶段，PM 可以快速模拟「任务从开始到结束」的流动，而不必在表格里逐个改字段。

---

### 6. Sprint 简报（`#/sprints`）

**目的**：按 Sprint 聚合任务进度与统计，用简报视图呈现迭代健康度（与优先级等字段联动）。

**用处**：适合在周会或迭代复盘场景中 **一页说清**：本 Sprint 里有哪些任务、完成度如何，而不是散落在多条任务记录里。

---

### 7. 任务规划（`#/planning`）

**目的**：用 **自然语言** 描述要做的任务，通过 **多轮对话** 澄清范围；必要时结合 LLM 做理解与结构化（无 API Key 时可降级演示）。

**用处**：解决「需求刚开始只有一段话」的现实情况：先把任务说清楚，再进入人选推荐，减少后续反复改方案的隐性成本。

---

### 8. 方案生成与对比（任务详情 → `#/tasks/:id/proposals`）

**目的**：在任务确定后生成 **多套人选组合方案**，附带匹配理由、团队适配评语、优势与风险；支持对方案的 **二次修改**，并保留历史版本。

**用处**：模拟真实立项评审——不是强行选一个算法最优解，而是给决策者 **可比对的选项** 与 **可追溯的版本**，便于对齐管理层与一线负责人。

**补充**：系统可对跨任务资源冲突做提示（例如某人已在另一任务中），降低「同一个人在两处被全职预订」的低级错误。

---

### 9. 任务回顾（`#/tasks/:id/review`）

**目的**：任务闭环后的 **结构化回顾**（执行情况、协作体感等），为后续能力更新提供依据。

**用处**：把复盘从聊天记录升级为 **可检索的记录**，并与下一模块衔接。

---

### 10. 能力值待审（`#/ability-updates`）

**目的**：基于方案修正、回顾内容等，自动生成 **能力值变更提案**；管理者可微调后 **一键应用**，形成模拟环境下的「反馈闭环」。

**用处**：体现产品设计主张：**用人结果应反哺对人的认知**。适合讨论绩效校准、晋升举证、培训需求等议题时使用沙盘数据演练流程。

---

### 11. 通知中心（`#/notifications`）

**目的**：把关键事件（如方案 finalize、回顾记录、能力值待审、进度变更等）收敛到 **统一收件箱**，避免信息散落在多个页面。

**用处**：PM 或管理者打开系统时能快速看到「有哪些事项需要处理」，贴近真实协作工具的习惯。

---

### 12. 自由对话（右下角「问问 AI」）

**目的**：提供全局 **问答与建议** 入口（默认偏只读认知；涉及写操作的意图需在界面二次确认），便于随手询问背景知识或梳理思路。

**用处**：降低「不知道怎么点菜单」的心智负担；与任务规划内的对话互为补充——前者偏全局助手，后者偏把某一个任务钉牢。

---

### 13. 顶栏工具（导出 / 导入 / 重置 / 全局搜索 / 主题）

**目的**：

- **导出 / 导入 / 重置**：保障 **`database.json` 单文件迁移**、演示环境与正式推演分离；写入前自动备份（详见 README）。
- **全局搜索（Ctrl/Cmd+K）**：跨员工、任务、技能、部门快速跳转。
- **深色模式**：长时间演示或分享屏幕时的可读性。

**用处**：PM 对外演示时可以「一键带回上次沙盘」；对内迭代时可以快速定位实体而不死记内部 ID。

---

## 可选的人工智能能力

系统在配置文件中接入 **任意兼容 OpenAI HTTP API** 的服务（如 DeepSeek、通义等）。若 **不配 API Key**，仍可运行 **离线 mock**，完整演示业务流程——便于采购前PoC、内网离线环境或对合规有顾虑时使用。

这说明：**智能化是加速器，不是门禁**。

---

## 适合谁、不适合谁（诚实边界）

**更适合**：需要做 **人选组合推演**、教学演练、方法论对齐的团队；希望有一套 **可解释、可复盘** 的沙盘，而不是黑盒「AI 说了算」。

**不太适合**：替代真实工时考勤、薪资、权限审计的生产 HR 系统；或需要多人实时协同编辑同一任务的云端 SaaS（当前定位为本地化模拟工具）。
