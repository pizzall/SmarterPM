# SmarterPM 产品介绍（面向产品经理 / 业务负责人）

本文面向 **产品经理、项目管理、人力资源伙伴或业务负责人**：不仅说明「有哪些界面」，还用 **业务语言** 解释 **背后的设计用意**，并把各模块涉及的 **字段与数值维度** 摊开来写清楚，便于对外宣讲或写立项材料。技术部署见 [DEPLOYMENT-PM.md](DEPLOYMENT-PM.md)，操作清单摘要见 [README.md](README.md)。

---

## 这套系统是干什么的（稍展开的电梯陈述）

SmarterPM 是一个跑在你自己电脑或内网服务器上的 **「公司执行模拟沙盘」**。你把组织架构和人放进系统里，用日常语言描述要做的事；系统会根据 **技能匹配、软性特质、角色分工、负载与成本** 等因素，给出 **多套人选组合**，并尽量用 **人话** 讲清「为什么这组人」「好在哪里」「风险在哪里」。任务走完后可以写 **回顾**，系统还能把磨合过程里暴露出的信息，整理成 **能力档案要不要改** 的待办，让「用人反馈」在模拟环境里有处可去。

它不打算替代真实的项目协作平台或人事主数据，更适合做 **方法论演练、立项前人岗辩论、培训演示**，或作为一种 **结构化、可解释的排兵布阵练习场**。

## 产品设计里坚持的几件事（刻意用非技术语言说）

下面这些不是实现细节，而是 **产品选择**，方便你和业务方对齐预期。

### 1. 「档案不全」不应把好人一棍子打死

真实世界里，很少有人事档案百分之百完整。若在模型里简单粗暴地排除缺失字段的人，会显得「死板」且脱离现实。

因此系统采用两层策略：

- **透明推断**：对部分缺失字段（例如沟通能力、MBTI、技能列表），会用已有信息 **估算**，并在界面上标明 **估算依据**（例如「用了部门同事的平均水平」「根据工作范围关键词猜了一个技能轮廓」）。
- **逐级放宽**：宁可先给一个「有据可查的猜测」，也不愿无声无息把人删掉——方便你在评审时追问「这个数字谁说了算」，而不是对着空白发呆。

这不是宣称推断永远准确，而是强调：**辅助系统要讲清楚自己凭什么说话**。

### 2. 「去哪里找人」跟着任务复杂度走

你可以把任务的 **复杂度** 理解成「这件事需要在多大的圈子里找人」：

- **普通**：优先在发起人所在部门 subtree 里找人——像是日常迭代里的常规搭配。
- **高级**：扩大到上一级部门——更像跨小组协作。
- **史诗**：允许在全公司候选池里找人——对应大型专项或全员动员级别的工作。

这样做的用意是：**不要让一个简单的活儿一上来就全公司海选**（噪声大、难解释），也不要让真正的大项目被部门墙限制死（错失关键人）。

### 3. 「推荐」其实是 **多方案 + 可读理由**

系统底层会给候选人算一个 **综合匹配度**，再按角色需求 **拼出 2～3 套不同风格的小队**。对业务方而言，这比只给「一个最优解」更接近真实决策：**比较、取舍、留下会议记录**，才是 PM 熟悉的节奏。

每套方案还会尝试给出 **团队适配评语、优势、风险、跨部门协作提醒**（在接入大模型时由模型润色；未接入时也有离线口吻的占位说明）。你可以把它理解成 **立项材料里的「人选方案对比页」自动生成草稿**。

### 4. **成本上限是「软约束」，不是硬砍人**

任务可以填一个 **预算上限**（`budget_cap`），员工档案里可以填 **成本率**（`cost_rate`，例如按人天费率理解即可）。系统会把预算按角色人数摊成「人均可用额度」，对 **明显高于** 该额度的人做 **降权或收窄候选池**，但 **不保证** 粗暴剔除所有人——避免因为数据填错一个零就全场无人。

业务语言：**预算是在提醒算法「优先找性价比合适的组合」，而不是替财务做最终裁决**。

### 5. **大模型是「加速器」，不是「大门」**

撰写任务澄清、润色方案文案、按自然语言调整方案等能力，在接入兼容 OpenAI 协议的 API 时会 **更顺滑**。若不配置密钥，系统会进入 **离线演示模式**：流程仍可走通，部分文案会变成规则化占位。

这保证 **内网演示、采购前沙盘、合规审慎** 的场景下，你仍能向干系人展示闭环，而不会被「必须先买 AI」卡住。

## 全局参考：枚举类维度与能力刻度（跨多个模块）

以下定义在界面里多以 **下拉框 + 中文说明** 呈现（便于非技术同事理解选项含义）。

### 任务与组织相关枚举

| 维度 key | 可选值（业务含义） |
|----------|-------------------|
| **任务复杂度 `complexity`** | **普通**：单部门可完成；**高级**：跨部门协作的复杂任务；**史诗**：全公司级大型任务。 |
| **任务状态 `status`** | **草稿**：尚未启动；**进行中**：已立项执行；**已完成**：已交付；**归档**：封存。 |
| **任务优先级 `priority`** | **低 / 普通 / 高 / 紧急**（用于看板排序、Sprint 简报视觉强调等）。 |
| **项目组状态** | **活跃 / 停用**。 |
| **回顾情绪 `mood`** | **积极 / 中性 / 消极**，帮助快速浏览回顾基调。 |

### 人员相关枚举

| 维度 key | 可选值（业务含义） |
|----------|-------------------|
| **角色倾向 `role_tendency`** | **领导者**：主导方向、把控质量；**执行者**：落实具体产出；**评审者**：把关质量与风险。用于 **推荐时更偏哪类席位**。 |
| **绩效趋势 `performance_trend`** | **上升 / 稳定 / 下降**，在打分里对应小幅奖励或惩罚——表达「近况」而非盖棺定论。 |
| **MBTI** | 常见 16 型 + 简短性格描述标签，用于 **团队气质、沟通风格** 的旁白（可空）。 |

### 能力值提案状态（能力值待审里）

| 状态 | 含义 |
|------|------|
| **待审** | 系统或流程刚生成建议，等你拍板 |
| **已修改** | 你动手微调了建议值 |
| **已应用** | 已写回员工档案 |
| **已拒绝** | 不接受该条建议 |

### 1～5 分能力刻度的中文标签（沟通、负责度、成长、技能等级等共用一套说法）

界面与元数据中会使用：**很差 → 较差 → 一般 → 不错 → 很好** 五档语义，对应数值 **1～5**（技能条上的 `level` 也按此理解）。这样 PM 在评审时有 **统一话术**，避免「3 分到底是好是坏」争论不一。

## 数据从哪进、从哪出（和模块的关系）

所有业务实体最终落在项目根目录的 **`database.json`** 里。你可以把它想象成 **一个可带着走的沙盘快照**：复制文件即备份，放到新机器上即恢复环境。顶栏的 **导出 / 导入 / 重置** 只是把这个文件的管理变按钮化；每次写入前系统还会自动滚存历史备份。

下面按 **业务模块** 说明：模块目的、典型工作流、**包含哪些字段与数值**，以及 **对 PM 有什么用**。

---

## 模块一：组织管理（`#/org`）

### 用来解决什么问题

把公司拆成 **树状部门**，并维护 **虚拟项目组**。没有这一层，后面「在谁的范围里找人」会失去参照系。

### 部门树里都有哪些信息

| 字段 | 类型/说明 | PM 可以怎么理解 |
|------|-----------|-----------------|
| `id` | 系统内唯一编号 | 内部引用，一般由系统生成或可人工指定 |
| `name` | 部门名称 | 与真实架构对齐时用 |
| `head` | 部门负责人（员工 id，可空） | 用于组织关系展示，可与推荐发起人形成故事线 |
| `parent_id` / `children` | 树结构 | 表达上下级关系；**推荐搜索范围**会沿这棵树行走 |
| `company` 根节点 | 内置顶层 | 下面是各业务单元 |

### 项目组里都有哪些信息

| 字段 | 说明 |
|------|------|
| `name` | 项目组名称，如「跨端重构项目组」 |
| `head` | 项目组负责人 |
| `members` | 核心成员列表（员工 id） |
| `status` | **活跃 / 停用** |

### 设计用意（白话）

部门树不是为了画漂亮图，而是为了在 **「普通 / 高级 / 史诗」** 复杂度下 **自动决定「去哪片池子捞人」**，让推荐结果 **和规模感一致**。

---

## 模块二：人员档案（`#/employees`）

### 用来解决什么问题

把每个员工作为 **候选人单元** 建档：谁能干什么、个性与行为倾向如何、的成本与近况如何。这里是 **整个人岗匹配的信息母版**。

### 人员档案包含的维度一览

**必填（写入时）**

| 字段 | 说明 |
|------|------|
| `name` | 显示名 |
| `departments` | 所属部门 id 列表（可跨部门兼职） |

**技能（列表 `skills`）**

| 子字段 | 类型 | 说明 |
|--------|------|------|
| `tag` | 文本 | 技能名称，来自或汇入「技能库」 |
| `level` | 数值，通常 **1～5** | 熟练度，对应「很差～很好」话术 |

**工作范围 `work_scope`**

- 文本标签列表，如「后端服务」「需求评审」。
- 当 **没有技能条** 时，系统可据关键词 **猜一套默认等级约 3 分的技能轮廓**（界面上会告诉你这是推断来的）。

**角色与风格**

| 字段 | 枚举/类型 | 用途 |
|------|-----------|------|
| `role_tendency` | 领导者 / 执行者 / 评审者 | 决定 **更擅长扛哪类席位** |
| `mbti` | 16 型之一或空 | 团队气质旁白；空时可用 **角色倾向 + 沟通分** 粗估占位（会标注来源） |

**软性数值（多为 1～5，可空）**

| 字段 | PM 理解方式 |
|------|-------------|
| `communication` | 沟通协作顺畅度 |
| `responsibility` | 交付责任与靠谱程度 |
| `growth_rate` | 成长速度/学习敏锐度 |

**近况与备注**

| 字段 | 说明 |
|------|------|
| `performance_trend` | 上升 / 稳定 / 下降，轻微影响总分 |
| `collaboration_notes` | 协作侧记（列表） |
| `special_notes` | 自由备注 |
| `cost_rate` | **成本率**（可选）；有 **任务预算上限** 时参与软约束 |

**系统维护类（了解即可）**

| 字段 | 说明 |
|------|------|
| `correction_log` | 历史上对档案字段的修改记录 |
| `_inferred`（接口返回） | 哪些值是原文如此、哪些是 **推算**、依据是什么 |

### 设计用意（白话）

- **技能 + 范围** 解决「会不会干」；**沟通、负责、成长** 解决「好不好一起干」；**角色倾向** 解决「适合站哪个位置」。
- **故意允许空值**：模拟「资料不全的真实公司」，同时用 **标注来源的推断** 避免黑箱。
- **成本率 + 任务预算**：让沙盘可以讨论「在钱/人力包有限时怎么组队」——但产品选择是 **软约束**，不把故事讲死。

## 模块三：人员状态（`#/people-status`）

### 用来解决什么问题

从 **「档案里的静态人」** 切换到 **「此刻活都在谁身上」**：谁空、谁正常忙、谁可能过载。

### 展示与聚合维度（由系统计算）

| 维度 | 说明 |
|------|------|
| `active_task_count` | 该人在 **进行中（active）** 任务里、被 **已定方案** 点名的次数 |
| `active_tasks` | 任务 id、标题、人在方案中的 **角色**、可选 **进度 %** |
| `level` | **空闲 / 常规 / 过载** 三档色标（由并发任务数等规则推断） |

### 设计用意（白话）

**加人之前先看热度图**，比只看「他技能对口吗」更接近真实 PM 的口头禅：「这人已经三块皮了，还能接吗？」

## 模块四：任务（`#/tasks` 及任务详情）

### 用来解决什么问题

把一件事变成 **结构化需求包**：给谁做、多难、多长、要什么技能与角色配比、跟钱和进度的关系是什么。任务对象会串联 **规划 → 方案 → 回顾 → 能力变更**。

### 任务字段与数值维度

**基础描述**

| 字段 | 说明 |
|------|------|
| `title` | 任务标题 |
| `description` | 详细描述（规划阶段与方案旁白都会引用） |
| `requester` | 发起人（员工 id，影响 **「从哪个部门出发画搜索圈」**） |

**规模与节奏**

| 字段 | 类型 | 说明 |
|------|------|------|
| `complexity` | 枚举 | 普通 / 高级 / 史诗 → 绑在 **找人范围策略** 上 |
| `duration_weeks` | 整数周 | 周期预期 |
| `sprint_id` | 可选 | 归属迭代，驱动 **Sprint 简报** |
| `status` | 枚举 | 草稿 / 进行中 / 已完成 / 归档 |
| `priority` | 枚举 | 低 / 普通 / 高 / 紧急 |

**需要怎样的人（硬性输入给推荐引擎）**

| 字段 | 说明 |
|------|------|
| `required_skills` | 关键词列表（与人员技能、工作范围匹配） |
| `required_roles` | 字典：如 `leader:1, executor:3, reviewer:1` → **每种角色要几个头** |

**成本与执行进展（偏管理驾驶舱）**

| 字段 | 说明 |
|------|------|
| `budget_cap` | 任务级 **预算上限**（与人员 `cost_rate` 联动，**软约束**） |
| `progress` | **0～100** 进度百分比（看板、人员状态卡片会引用） |
| `blockers` | 阻塞项列表 |
| `milestones` | 里程碑子表：每项含 `title`、`due`（日期）、`done`（是否完成） |

**任务之间的关系**

| 字段 | 说明 |
|------|------|
| `depends_on` | 依赖的其他任务 id（表达先后关系） |
| `primary_task` / `sub_tasks` | 父子任务 linkage（大块工作拆分子工作时） |

**与方案、回顾挂载的数据（在任务 JSON 内部）**

| 字段 | 说明 |
|------|------|
| `proposals` | 多套方案及历史修改（见下一节） |
| `review` | 回顾记录列表（见回顾模块） |

### 设计用意（白话）

任务对象刻意 **把「事」与「人」绑在一起**：事的难度决定 **圈人的半径**，事的角色配比决定 **小队编制**，事的预算讨论 **性价比**，事的进度与阻塞 **反哺「人员状态」与通知**——这是一套讲故事用得上的数据结构。

## 模块五：看板（`#/board`）

### 用来解决什么问题

用 **拖拽** 改变任务 **状态**，模拟迭代例会上的「从这列挪到那列」。

### 涉及的维度

实质仍然是任务的 **`status`** 四枚举；可选展示 **`priority`** 色标等。

### 设计用意（白话）

降低「为了改状态点进多层表单」的摩擦，让沙盘演示 **更像在看一块物理白板**。

---

## 模块六：Sprint 简报（`#/sprints`）

### 用来解决什么问题

把一个 **时间盒迭代（Sprint）** 里的多个任务 **收成一页简报**：什么时候开始、多长、底下挂了哪些任务、进度分布如何。

### Sprint 实体字段（库内 `sprints`）

| 字段 | 说明 |
|------|------|
| `sprint_id` | 迭代编号 |
| `start_date` | 开始日期 |
| `duration_weeks` | 持续周数 |
| `tasks` | 本迭代包含的任务 id 列表 |

任务的 **`priority`、`progress`、`status`** 会参与简报层的排序与观感。

### 设计用意（白话）

当你要向管理层解释「这两周我们在干嘛」时，简报视图回答的是 **迭代级别的叙事**，而不是单个任务的琐碎字段。

---

## 模块七：任务规划（`#/planning`）

### 用来解决什么问题

把 **一段话的需求** 逐步磨成 **结构化任务草稿**：标题、描述、复杂度、周期、所需技能、角色配比，以及 **尚需澄清的问题列表**。

### 对话流里的产物（存入会话草稿）

典型草稿字段与任务模型对齐，包括但不限于：`title`、`description`、`required_skills`、`required_roles`、`complexity`、`duration_weeks`、`clarifying_questions`（系统建议你追问业务方的条目）。

### 设计用意（白话）

模仿真实立项：**第一次开口往往不完整**，允许 **多轮澄清** 比「一次性表单填完」更符合工作坊节奏。若启用大模型，由模型帮你 **揪缺口**；关掉模型时也有 **关键词启发式** 保底。

---

## 模块八：人选方案（任务详情 → `#/tasks/:id/proposals`）

### 用来解决什么问题

对已定稿的任务 **一键生成多套人选组合**，每套包含：

| 区块 | 业务含义 |
|------|-----------|
| **成员列表 `members`** | 每位员工在当前方案里的 **席位角色**，以及 **匹配理由短语** |
| **`team_fit`** | 把小队放在一起时的 **气质与协作观感** |
| **`advantages`** | **优势叙事** |
| **`risks`** | **风险与提醒** |
| **`cross_dept_notes`** | **跨部门时要特别注意什么** |

每条方案还有 **`version`**（随修改递增）、**`modifications`**（每次指令修改前的快照 + 修改说明）。

### 底层打分维度（不必背公式，但要理解「系统在算什么」）

对每个候选人，引擎会综合：

| 维度 | 白话 |
|------|------|
| **技能匹配 `skill_score`** |  Required 技能 vs 档案技能等级 |
| **工作范围重合 `scope_score`** | Required 关键词是否出现在日常职责描述里 |
| **软性均值 `soft_score`** | 沟通、负责、成长的组合；缺的可用 **部门均值** 顶替（会备注） |
| **角色契合 `role_fit`** | 领导者 / 执行者 / 评审者 **席位加权** |
| **绩效趋势奖惩** | 近期上行小幅加分，下行小幅减分 |

再加 **多套组合之间的差异化策略**（例如第二套有意换血主要成员），避免三套方案听起来像复制粘贴。

### 资源冲突提示（关联 API）

生成或审阅时可结合 **跨任务冲突检测**（某人是否已在其他 **进行中** 任务里占用），用业务语言提醒 **「人手已经排满」**，减少沙盘里出现「同一人在两个战场都是全职主角」的尴尬故事。

### 设计用意（白话）

- **多套方案**：服务 **评审与政治过程**，不是替代负责人拍板。
- **解释性成员理由 + 段落化优劣势**：方便 **直接粘贴进会议纪要** 或立项邮件做初稿。
- **版本与修改链**：对应真实世界里「上次定的班子再微调一版」的可追溯性。

## 模块九：任务回顾（`#/tasks/:id/review`）

### 用来解决什么问题

给每条任务提供 **复盘条目**：谁在什么时间写了什么、情绪基调如何。

### 字段

| 字段 | 说明 |
|------|------|
| `content` | 回顾正文 |
| `author` | 作者（员工 id，可空） |
| `mood` | **积极 / 中性 / 消极** |

### 设计用意（白话）

把「事后感觉是靠谱还是不靠谱」 **落成记录**，后续才能严肃讨论 **要不要调整对人的看法**——而不是停在饭桌上的一句闲话。

## 模块十：能力值待审（`#/ability-updates`）

### 用来解决什么问题

当 **方案被修改**、或 **回顾提交了文本** 等触发条件满足时，系统尝试自动生成若干 **档案字段变更提案**（例如某项软性分数微调、`performance_trend` 枚举变更等）。你在本模块 **审核、微调、应用或驳回**。

### 提案常见属性（概念层面）

| 概念 | 说明 |
|------|------|
| 针对的员工 | 哪位同事的哪个字段 |
| 建议值 vs 当前值 | 改了什么 |
| 依据说明 | 引用触发文本或摘要 |
| 状态 | 待审 / 已修改 / 已应用 / 已拒绝 |

系统在生成时会遵守 **单次小幅调整**、**不轻易瞎编** 等产品约束（详见 prompts 规则），本质是 **帮你起草「校准档案」的流程工单**，而不是偷偷重写所有人。

### 设计用意（白话）

用人结果是 **持续的反馈**，档案应是 **活的**。沙盘里我们把这条链路做成 **显式工单**，让你可以和 HR / 组长一起讨论 **哪些自动建议采纳、哪些不当真**。

## 模块十一：通知中心（`#/notifications`）

### 用来解决什么问题

把散落在各处的 **「有待你处理的事」** 收到一个收件箱：例如任务状态变更、方案已定稿、回顾刚提交、能力提案排队中等。

### 通知类型举例（概念）

包括但不限于：**任务创建 / 状态变更 / 方案落定或修改 / 新回顾 / 能力待审 / 进度更新 / 通用提示**。每条可有标题、正文摘要、跳转链接、关联实体 id。

### 设计用意（白话）

沙盘演示时观众注意力有限——**铃铛红点**比「请记住刚才改了哪个深层页面」更符合人性。

## 模块十二：自由对话（右下角「问问 AI」）

### 用来解决什么问题

提供一个 **全局问答与 brainstorm 抽屉**：随时提问、请系统梳理上下文或给建议。

### 维度

主要是 **会话消息流**（用户输入与助手回复），带历史会话切换；涉及 **真实写入数据库的操作** 时，界面会要求 **二次确认**，避免演示时一句话误删数据。

### 设计用意（白话）

把它当成 **不占版面的参谋**：不必离开当前页面去搜索引擎；但对 **改动现实的动作** 保留护栏。

## 模块十三：顶栏全局能力（搜索 / 主题 / 数据库）

| 能力 | PM 视角的用处 |
|------|----------------|
| **全局搜索（Ctrl/Cmd+K）** | 跨员工、任务、技能、部门 **跳转**，演示时不死记内部编号 |

## 什么时候很适合用、什么时候不适合（边界再次说清楚）

**很适合**：需要做 **结构化人选推演**、对内对齐「什么样的人适合扛哪类活」、做培训或方法论工作坊。

**不适合**：替代真实考勤薪资系统；也不适合默认等同于「公司已正式采纳的人事结论」——沙盘的价值在于 **对话与举证**，而非一纸公文。
